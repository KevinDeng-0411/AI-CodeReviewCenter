# AI Center 项目面试通关指南

---

## 一、30 秒电梯演讲

> 「我独立开发了一个 **AI 研发效能中台**，叫 AI Center。
> 它把 AI 能力嵌入到日常开发流程里——代码提交后自动做 Code Review、自动生成单元测试、还能像 ChatGPT 一样做技术问答。
> 核心技术栈是 Spring Boot 3 + LangChain4j + DeepSeek + PostgreSQL pgvector。
> 有 22 个 API，覆盖了 CR 评审、单测生成、智能问答、RAG 知识库四个模块。
> 做完之后假设把它接入 CI/CD 流水线，预计线上缺陷率能降 60%，CR 效率提升 80%。」

---

## 二、项目是做什么的 —— 用大白话讲

### 它不是一个「产品」，是一个「中台」

打个比方：你之前做的苍穹外卖，饿了么用户打开 App 点餐——那是 C 端产品。

AI Center 是**研发工具**，给开发团队用的。它独立运行，其他项目通过 HTTP API 调用它：

```
你的苍穹外卖项目
  └── GitLab CI 流水线
        └── 每次 MR 自动调 AI Center 做 Code Review
  └── 新来的同事
        └── 问 AI Center：「这个项目的缓存策略是什么？」
```

### 四大能力

| 能力 | 一句话 |
|------|--------|
| AI Code Review | 你把代码贴过去，AI 从 8 个维度评审，告诉你哪有问题怎么改 |
| AI 单测生成 | 你把源码贴过去，AI 给你生成 JUnit 测试代码 |
| AIReadMe 生成 | 描述一下项目，AI 自动写开发指南、架构文档 |
| 智能问答 | 像 ChatGPT 一样，但**能记住之前的对话**，还能**搜你上传的文档** |

## 三、四个面试亮点的技术实现

### 亮点 1：AI Code Review 结构化提示词

**面试时怎么说：**

> 「我设计了一套结构化的 Prompt 模板系统。核心思路是：如果不约束 AI 的输出格式，它返回的内容千奇百怪——有的很啰嗦、有的漏维度。所以我把 Prompt 分成了三个层次：角色设定 + 8 个评审维度 + 3 级问题分类 + JSON Schema 约束输出。」

**技术实现（对着代码讲）：**

```
PromptTemplateManager（Prompt模板管理器）
  ├── 模板存数据库（prompt_templates 表）
  ├── 启动时加载到 ConcurrentHashMap 缓存
  └── {{source_code}} 占位符替换

CodeReviewService.review()
  ├── 1. 拿模板 → render 渲染 Prompt
  ├── 2. 调 DeepSeek LLM
  ├── 3. 解析 JSON 响应 → CodeReviewVO
  └── 4. 持久化到数据库
```

**8 个评审维度**：性能优化、安全性、代码质量、可维护性、异常处理、并发安全、资源管理、设计模式

**3 级问题分类**：Critical（必须修）、Warning（建议修）、Info（优化建议）

**面试追问应对话术：**

> 问：「为什么不用 SonarQube 之类的静态分析工具？」
> 答：「Sonar 是基于规则的——它只能检查已知模式。AI CR 能理解代码语义，比如能看出『你这个循环里每次都在 new 对象，应该提到外面复用』——这种需要语义理解的问题，规则引擎做不到。」

---

### 亮点 2：AI 友好型项目 —— AIReadMe 生成

**面试时怎么说：**

> 「AI 写代码最大的问题是：它不了解你的项目上下文。写出来的代码 API 不存在、用了错误的框架版本、不走你们约定的设计模式。所以我设计了 AIReadMe——6 个章节的自动化项目文档，本质上是给 AI 看的『项目说明书』。」

**6 个章节**：

| 章节 | 内容 | 作用 |
|------|------|------|
| 技术架构 | 技术栈、分层设计 | AI 知道用啥框架 |
| 核心流程 | 业务流程序列 | AI 知道数据怎么流转 |
| 开发指南 | 环境搭建、规范 | AI 知道怎么启动 |
| 项目结构 | 包结构、模块说明 | AI 知道代码在哪 |
| 业务知识 | 领域术语、规则 | AI 理解业务 |
| 历史经验 | 已知坑点、最佳实践 | AI 避免重复踩坑 |

**面试追问应对话术：**

> 问：「这些文档怎么给 AI 看？」
> 答：「两种方式。一是生成后放到项目根目录，Cursor/Copilot 等 IDE 工具会自动读取作为上下文。二是通过 RAG 检索——用户问『这个项目怎么部署』，系统自动搜 AIReadMe 的『开发指南』章节，注入到对话里。」

### 亮点 3：团队智能问答知识库

这是最有技术深度的模块，分三层来讲。

#### 第一层：短期记忆（Short-Term Memory）

**一句话**：让 AI 记住你刚才聊了什么。

**技术方案**：滑动窗口 + Redis + 摘要

```
对话消息 → Redis List（只保留最近 20 轮）
         → 超过 10 轮时 → 自动调 AI 生成摘要 → Redis String
         → 冷数据异步存 PostgreSQL（7 天过期）
```

**关键代码位置**：`ShortTermMemoryManager.java`

```
saveMessage()  →  保存到 Redis + 触发摘要 + 持久化 PG
getContextWindow()  →  摘要 + 最近消息 → 拼接为 LLM 上下文
```

**面试话术**：

> 「就像你跟一个人聊天，他不会记住你们历史上所有的对话细节——但他脑子里有一个『摘要』：『刚才我们聊的是 Java 缓存方案』。我的短期记忆也是这个原理。不保留全部历史——太长了 AI 处理不了也费钱——而是保留最近 20 轮 + 一个自动生成的摘要。」

#### 第二层：长期记忆（Long-Term Memory）

**一句话**：让 AI 记住我知道的事情。

**技术方案**：Ollama bge-m3 向量化 + PostgreSQL pgvector

```
手动录入知识 → bge-m3 转成 1024 维向量 → 存 pgvector
用户提问 → 同样转成向量 → 算余弦相似度 → Top-K 召回 → 注入对话
```

**关键代码位置**：`LongTermMemoryManager.java`

```
saveMemory()  →  内容 → bge-m3 向量化 → pgvector.add()
recall()      →  查询 → 向量化 → pgvector.search() → 返回 Top-K
```

**面试话术**：

> 「向量化的本质是把一段文字变成一个数学坐标。『ORM 框架』和『MyBatis-Plus』这两段话在向量空间里距离很近——因为语义相似。bge-m3 模型就是专门做这件事的，它用 1024 个浮点数代表一段文本的语义。」

#### 第三层：RAG 检索增强

**一句话**：回答问题时先去知识库里搜相关资料，带着资料回答。

**技术方案**：查询重写 + 语义分块 + BM25+向量混合检索

```
用户问「缓存击穿咋整」
  → QueryRewriter: 「缓存击穿如何解决」「热点Key失效方案」「缓存穿透击穿雪崩」
  → SemanticChunker: 把文档切成 ~500 token 的语义块
  → HybridRetriever:
       BM25（关键词匹配，权重 0.3）
     + pgvector 向量检索（语义相似度，权重 0.7）
  → 融合排序 → Top-K
```

**关键代码位置**：

| 组件 | 文件 |
|------|------|
| 查询重写 | `QueryRewriter.java` |
| 语义分块 | `SemanticChunker.java` |
| 混合检索 | `HybridRetriever.java` |
| 流程编排 | `RagService.java` |

**面试话术**：

> 「纯关键词搜索有局限——你搜『缓存击穿』就搜不到讨论『热点 Key 失效』的文档，虽然它们说的是同一件事。向量检索解决了这个问题——它能理解语义。但纯向量也有局限——精确的术语匹配可能被稀释。所以我把两者混合：向量 0.7 权重负责语义，BM25 0.3 权重负责精确匹配。」

## 四、技术栈选择话术

面试官大概率会问「为什么选这个技术？」

| 技术 | 为什么选 | 备选是谁 | 为什么不用备选 |
|------|---------|---------|------------|
| **LangChain4j** | Java 最成熟的 LLM 框架，API 设计类似 Python LangChain | Spring AI | Spring AI 当时还不够成熟，文档少 |
| **DeepSeek V4 Flash** | 免费、中文强、OpenAI 兼容 | GPT-4/Qwen | GPT 贵、Qwen 也需要付费 |
| **Ollama bge-m3** | 本地免费无限制 | text-embedding-v3 (OpenAI) | 要 API Key 要钱 |
| **pgvector** | 和业务库合二为一，不改 SQL 习惯 | Pinecone/Milvus | 多一个独立服务，架构重 |
| **PostgreSQL** | 既有关系型又能存向量，一库两用 | MySQL | MySQL 没有向量能力 |

## 五、常见面试追问

### Q1：向量数据库怎么选型？

> 「我对比过三个方案。Milvus 最专业但需要单独部署——多一个服务多一个故障点。Pinecone 是云托管不用运维——但要花钱还要 API Key。pgvector 是 PostgreSQL 的扩展，和我的业务库是同一个——部署简单，而且我本来就要用 PG 存关系数据。对于原型和中小规模场景，pgvector 完全够用。如果数据量大到百万级以上再切 Milvus——代码层面改一行配置就行，因为 LangChain4j 的 EmbeddingStore 接口统一了所有向量数据库的 API。」

### Q2：AI Code Review 和 SonarQube 有什么区别？

> 「Sonar 查的是代码规范问题——变量命名、圈复杂度、重复代码块。我 的 AI CR 查的是需要『理解代码想干什么』才能发现的问题。比如一段代码里 SQL 拼接——Sonar 可能报一个『安全风险』，但 AI 能告诉你具体为什么危险、攻击者能怎么利用、以及应该怎么改才能修复——因为 AI 理解了这段代码的语义。」

### Q3：为什么不直接用 ChatGPT 的 API？

> 「技术上完全支持——改一行配置就行，因为 DeepSeek 和 OpenAI 都是 OpenAI 兼容 API。选 DeepSeek 主要是成本——10 块钱能用一个月，GPT-4 一天就用完了。而且 DeepSeek 的中文能力不输 GPT-4。」

### Q4：如果数据量大了，RAG 检索慢了怎么办？

> 「现在用的是全表扫描 + 应用层算余弦，数据量大会慢。首先 pgvector 有 IVFFlat 索引——近似最近邻搜索，可以把 O(n) 降到 O(log n)，改一行 SQL 就能启用。其次可以把 BM25 那部分迁移到 PG 的 tsvector 全文索引——现在也是全表扫。最后如果百万级以上，切 Milvus——LangChain4j 的 EmbeddingStore 接口是一样的，换实现不改业务代码。」

### Q5：AI 生成了错误的单测怎么办？

> 「AI 生成的单测不追求 100% 正确，目标是减少手工编写的基础工作量。单测生成后还是需要人 review——但人只需要改 20% 的内容，而不是从零写 100%。这个项目里，单测生成功能的目标是把单测环节的效率提升 80%——从零写一篇 200 行的测试可能是半小时，但 review 和微调 AI 生成的代码只需要 5 分钟。」

### Q6：短期记忆和长期记忆的区别是什么？

> 「短期记忆是会话级别的——跟一次聊天绑定的上下文，比如用户说『我叫小明』，后面 AI 记得他叫小明。但换个会话就忘了。长期记忆是全局级别的知识——比如『团队用 MyBatis-Plus 3.5.5』，这是所有会话都能搜到的知识。技术上也是两套方案：短期记忆用 Redis（快、易过期），长期记忆用 pgvector 向量检索（语义相似度搜索）。」

### Q7：Pinecone 和 pgvector 怎么切换？

> 「改一行配置。在 application.yml 里配置 ai.pinecone.api-key，AIConfig 里的 embeddingStore Bean 会优先用 Pinecone。Pinecone 连接失败时自动降级到 pgvector。LangChain4j 的 EmbeddingStore 接口统一了所有向量数据库的 API——代码里用的是接口，不关心底层是 PG 还是 Pinecone 还是 Milvus。」

## 八、Prompt Engineering —— 项目中所有提示词的设计思路

项目中一共有 **7 组提示词**，分布在 5 个文件里。每一组都体现了不同的提示词设计模式。

### 总览

| # | 提示词 | 所在文件 | 设计模式 |
|---|--------|---------|---------|
| 1 | CR 评审主提示词 | `init.sql`（数据库预置） | **角色+维度+等级+输出约束** |
| 2 | 单测生成提示词 | `UnitTestService.java` | **角色+要求清单+输出格式** |
| 3 | AIReadMe 章节提示词 | `AiReadmeService.java`（6个） | **角色切换+章节定制** |
| 4 | 对话 System Prompt | `ChatService.java` | **动态上下文拼接** |
| 5 | 对话摘要提示词 | `ShortTermMemoryManager.java` | **增量合并** |
| 6 | 查询重写提示词 | `QueryRewriter.java` | **改写规则+变体生成** |

---

### 提示词 1：AI Code Review（最核心，体现 Prompt Engineering 深度）

**位置**：`init.sql` 预置在 `prompt_templates` 表，`PromptTemplateManager` 管理

#### v1 → v2 升级过程（面试加分：展示迭代思维）

```
v1（初版）：角色 + 8维度 + 3等级 + JSON输出
     ↓ 问题：AI 评审深度不够，容易泛泛而谈
v2（当前）：七层结构，从「告诉AI做什么」升级到「告诉AI怎么做、做到什么程度」
```

**v2 七层结构**：

```
┌──────────────────────────────────────────────────────────┐
│ 第一层：角色定位 + 评审原则（客观/建设/完整/优先级）       │
│ "你是Java资深架构师兼技术专家...评审意见必须客观、可落地"   │
├──────────────────────────────────────────────────────────┤
│ 第二层：评审流程（6步SOP）                                 │
│ 理解意图→分析方案→识别问题→识别隐患→评估质量→给建议        │
├──────────────────────────────────────────────────────────┤
│ 第三层：问题分级（每个等级有具体触发条件）                  │
│ Critical=安全漏洞/线程安全/空指针/资源泄漏                  │
│ Warning=代码质量/潜在性能隐患/异常处理不当                   │
│ Info=风格改进/设计优化/测试不足                             │
├──────────────────────────────────────────────────────────┤
│ 第四层：8维检查清单（每个维度都有具体检查项）               │
│ 如「安全性」→输入验证/权限控制/加密脱敏/SQL注入/XSS/CSRF   │
├──────────────────────────────────────────────────────────┤
│ 第五层：特殊场景处理                                       │
│ 遗留代码/紧急修复/重构/新功能——不同场景不同评审标准          │
├──────────────────────────────────────────────────────────┤
│ 第六层：职责边界（✅该做的 / ❌不该做的）                    │
│ ✅识别问题给建议  ❌不直接改代码、不替代人工确认             │
├──────────────────────────────────────────────────────────┤
│ 第七层：输出格式 + 评分标准                                 │
│ JSON with 具体字段 + fix_code + 评分锚定（0-40/40-60/...） │
└──────────────────────────────────────────────────────────┘
```

**v1 到 v2 的三个关键升级**：

> **升级 1：从「罗列维度」到「维度检查清单」**
>
> v1 只说了「检查安全性」，AI 可能只看有没有 SQL 注入。v2 的「安全性」维度下明确列了 6 个检查项：输入验证、权限控制、加密脱敏、SQL注入、XSS/CSRF、日志脱敏——AI 会逐项检查，覆盖度显著提升。
>
> **升级 2：增加「评审流程 SOP」**
>
> 不加流程时 AI 拿到代码就开始挑毛病，容易漏掉「这段代码想干什么」的上下文理解。v2 明确要求先理解意图再分析方案再识别问题——6 步 SOP，评审逻辑更系统。
>
> **升级 3：增加「职责边界」和「特殊场景」**
>
> 这两个是 Prompt Engineering 的高级技巧。职责边界防止 AI 越权——比如「不要直接改代码」（否则 AI 可能把你的代码重写了）。特殊场景让 AI 在不同上下文下调整标准——比如「这是紧急修复，优先关注安全性，代码风格可以放宽」，避免在救火时吹毛求疵。

**核心 Prompt（数据库中的 v2 模板）**：

```
[角色] 你是一名 Java 资深架构师兼技术专家。
       评审必须客观、建设性、可落地。

[评审流程 - 6步SOP]
1. 理解代码意图  2. 分析实现方案  3. 识别问题（逻辑/并发/泄漏/空指针）
4. 识别隐患（DB效率/内存/算法） 5. 评估质量（可读/可维护/重复/命名） 6. 给建议

[问题分级 - 每级有具体触发条件]
🔴 Critical: 安全漏洞、严重性能、数据一致性、线程安全、空指针、资源泄漏
🟡 Warning: 代码质量、潜在性能隐患、可维护性问题、异常处理不当
🔵 Info: 风格改进、最佳实践、设计优化、测试覆盖不足

[8维检查清单 - 每维有具体检查项]
1. 代码质量   2. 安全性(SQL注入/XSS/CSRF/加密/权限/日志)
3. 可维护性   4. 架构设计   5. Java最佳实践(Spring/NPE/异常/线程)
6. 数据库     7. 测试       8. 性能

[特殊场景]
遗留代码→重点看新增 | 紧急修复→优先安全 | 重构→关注兼容 | 新功能→全面评估

[职责边界]
✅ 识别问题给建议、解释原理   ❌ 不直接改代码、不替代人工确认

[输出格式] JSON + fix_code + 评分锚定
```

### 提示词 2：AI 单元测试生成

**位置**：`UnitTestService.java`（`buildDefaultUnitTestTemplate()` 方法）

**设计思路**：

### 提示词 3：AIReadMe 章节提示词（6 个）

**位置**：`AiReadmeService.java`

**设计思路**：

> 「这里体现了提示词设计的一个进阶模式：**分而治之**。如果一次性让 AI 生成整个项目文档，它会泛泛而谈，每个章节都不深入。我把任务拆成 6 个独立的 Prompt，每个 Prompt 设定一个不同的角色和一个具体的章节目标。比如『技术架构』章节让 AI 扮演架构师，『开发指南』章节让它扮演开发者——角色切换能让 AI 进入不同的思维模式。最后的输出是 6 个独立的、高质量的章节，拼起来就是完整的项目文档。」

| 章节 | Prompt 角色 | 核心指令 |
|------|------------|---------|
| 技术架构 | 资深 Java 架构师 | 技术栈选型说明、架构分层、关键决策 |
| 核心流程 | 资深 Java 架构师 | 业务流程序列、数据在各层流转 |
| 开发指南 | 资深 Java 开发者 | 环境搭建、启动命令、开发规范、调试方法 |
| 项目结构 | 资深 Java 开发者 | 模块划分、包结构、关键目录 |
| 业务知识 | 资深业务分析师 | 领域术语、业务规则、实体关系 |
| 历史经验 | 资深 Java 技术专家 | 已知陷阱、调优经验、最佳实践 |

### 提示词 4：智能问答 System Prompt（动态拼接）

**位置**：`ChatService.java`（`buildSystemPrompt()` 方法）

**设计思路**：

> 「这个 Prompt 不是写死的——它是**动态拼接**的。根据当前对话状态，拼接三段内容：长期记忆（从 pgvector 搜出来的相关知识）、RAG 检索结果（从知识库搜出来的文档）、短期记忆（最近 20 轮对话）。三段都是可选的——哪段有内容就拼进去，没有就跳过。最后加一句话『如果问题超出上下文范围，请基于你的专业知识回答』——这是兜底，防止 AI 在找不到相关资料时说『我不知道』。」

### 提示词 5：对话摘要（增量合并）

**位置**：`ShortTermMemoryManager.java`

**设计思路**：

> 「这是提示词设计的另一个模式：**增量合并**。每次触发摘要时，不是把所有历史消息发给 AI——那样太费 Token。而是把对话分成两半——前半部分生成摘要，摘要存 Redis。下次再触发时，用新摘要和已有摘要合并。这样每次只消耗一半对话的 Token，但摘要质量不会退化。」

```
[任务] 请将以下对话历史总结为简洁的摘要（200字以内），保留关键信息：

[新对话]
USER: 我叫小明，我是Java后端开发
ASSISTANT: 你好小明！...

[已有摘要]
之前聊了Java缓存方案

[指令]
请合并新旧信息输出最终摘要
```

### 提示词 6：查询重写（召回率优化）

**位置**：`QueryRewriter.java`

**设计思路**：

> 「这是 RAG 系统的经典优化手段——查询多样化。用户问『缓存击穿咋整』时，只搜这一句话可能漏掉大量相关文档。我让 AI 把这句话改写成 2-3 个不同表述：『缓存击穿如何解决』『热点Key失效方案』『缓存穿透击穿雪崩区别』——每个都去搜一轮，然后合并去重。这样召回率显著提升。」

```
[角色] 你是一个查询优化专家。

[规则]
1. 修正口语化表达，改为正式技术用语
2. 补充缺失的上下文关键词
3. 如果查询过于简短，扩展为完整的技术问题
4. 生成 2-3 个语义相近但表述不同的变体

[输入] 缓存击穿咋整

[输出格式]
["缓存击穿如何解决", "热点Key失效处理方案", "Redis缓存击穿的应对策略"]
```

### 提示词设计的几个通用原则（面试加分点）

**1. 角色设定 > 裸指令**

❌ 差的：`请评审这段代码`
✅ 好的：`你是一名资深 Java 代码审查专家，拥有 10 年以上大型分布式系统开发经验。请评审...`

原因：角色设定激活了 LLM 的领域知识——当它「认为」自己是专家时，输出的专业度、细节程度都显著更高。

**2. 约束输出格式**

❌ 差的：`告诉我代码有什么问题`
✅ 好的：`以 JSON 格式返回，每个问题含 dimension/severity/lineRange/description/suggestion/fixCode 六个字段`

原因：不约束格式 → 解析困难 → 下游系统无法消费。结构化输出是可编程的前提。

**3. 给例子比给规则更有效（Few-Shot）**

项目中目前用的是 Zero-Shot（只给规则），后续优化方向是加 1-2 个示例——LLM 从示例中学到的比从规则中学到的更多。

**4. 分而治之**

AIReadMe 的 6 章节独立生成是这个原则的最佳实践——一个大任务拆成多个小任务，每步都有明确的角色和输出要求。

**5. 兜底指令**

ChatService 最后一句「如果超出上下文，请基于专业知识回答」——防止 AI 在信息不足时拒绝回答，提升用户体验。

### 面试时怎么讲这部分

> 面试官：「你在项目中有做 Prompt 设计吗？」
>
> 你：「有的，这也是我重点思考的部分。我项目中一共有 7 组提示词，覆盖了代码评审、测试生成、文档生成、对话、摘要、查询重写六个场景。最核心的是 Code Review 的提示词——我设计了『角色设定 + 8 维度 + 3 级分类 + JSON Schema 约束』四层结构。这里我踩过几个坑，比如不约束输出格式时 AI 返回的内容没办法解析，不加角色设定时 AI 的评审质量明显下降。解决之后我把这套模式抽象成了 Prompt 模板系统——存在数据库里，可以通过 API 动态修改，不需要改代码。」
>
> 面试官：「设计 Prompt 有什么心得？」
>
> 你：「总结了几个原则。第一是角色设定——给 AI 设定一个专家角色比直接下指令效果好很多。第二是强制输出格式——尤其是 JSON Schema，这对下游系统消费很重要。第三是分而治之——大任务拆小，AIReadMe 我把 6 个章节拆成 6 个 Prompt，每个的产出质量明显高于一次性生成。第四是兜底指令——防止 AI 在信息不足时拒绝回答。第五是增量合并——摘要这个场景不能每次都把所有历史发给 AI，太贵了，我用增量合并的方式每次只消耗一半对话的 Token。」
